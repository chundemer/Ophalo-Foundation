# Decision Index — OpHalo Foundation

The living ledger of architecture/product decisions for **this** build. One line per
decision. This file is authoritative for the new project.

> The legacy reference app has its own `decision-index` and `decisions/` — those
> contain many decisions that no longer apply and are **not** used here. Consult
> them only as historical reference (`_reference/docs/`), and only after validation.

## Conventions

- Add a one-line entry the moment a decision is made. Expand to a full ADR under
  `docs/decisions/ADR-NNN-*.md` only when the rationale needs more than a line.
- IDs are sequential and never reused. A reversed decision is marked **Superseded**
  (link the successor), not deleted.

**Status legend:** `Implemented` = decided and built/verified in this repo ·
`Locked` = decided (build plan §4), not yet built · `Proposed` = under discussion ·
`Superseded` = replaced.

| ADR-034 | _(Phase4c-004)_ `UserAccessPolicy.IsPermitted(role, membershipStatus, accountPurpose, key)` composes three gates: (1) only `MembershipStatus.Active` is permitted anything; (2) `internal.*` keys additionally require `AccountPurpose.Internal` — preserving the legacy `AdminGuard` semantic that internal authority comes from the account's purpose boundary, not a role name; (3) explicit role-map check. **Fail-closed**: empty/whitespace/unknown/non-canonical keys denied, no trimming. | ADR-035 | _(Phase4d-001)_ Slice scope = **§4.11 feature/limit layer**. | ADR-037 | _(Phase4d-003)_ Usage limits resolve as **plan default with per-account override**; in v1 only `account.user_limit` is overridable, via the existing stored `AccountEntitlements.MaxUserSeats` (`> 0` wins, else plan default). | ADR-038 | _(Phase4d-004)_ `FeatureAccessPolicy` is **fail-closed** (blank/unknown feature key → `false`; blank/unknown key or unmapped plan → limit `0`; exact match, no trimming) and **standalone** — no combined permitted+entitled facade until a real caller exists (composition at the call site: account allowed **and** user permitted **and** account entitled). | ADR-039 | _(Provisioning-001)_ Account-creation orchestration is a **pure domain provisioning service** (`AccountProvisioningService` in Foundation.Application) — composes the four Phase 4 factories into the canonical `User + Account + owner AccountUser + AccountEntitlements` graph and returns it as `Result<AccountProvisioningResult>`. **No repositories / `IUnitOfWork` / MediatR / command-handler / DI** are introduced: the slice answers "what graph should exist?", and the persistence seam (save contract, transaction, unique constraints, aggregate boundary) is deferred to the persistence phase to avoid early-abstraction churn at that exact seam. Uses `DateTime` (UTC) to match the entity factory contracts, not the sketch's `DateTimeOffset` | Implemented | build-log/007 |
| ADR-040 | _(Provisioning-002)_ Cross-aggregate provisioning rules are **fail-closed**: the internal-account triad (purpose `Internal` ⇒ plan `Internal`, no pilot, no trial window) and the non-internal trial-window requirement return `Result.Failure`; seat allowance is **plan-derived** (`PlanEntitlements.LimitDefaults[plan][account.user_limit]`, ADR-037) — the method takes **no** seat parameter, so account↔entitlements seat drift is unrepresentable. Per-field validity (non-empty name, UTC timestamps, IANA TZ) is trusted to the entity factories (which throw); `Result.Failure` is reserved for composition rules. Primary owner is assigned in-memory pre-persistence (IDs exist at construction, ADR-019/024) and its failure is propagated. Provisioning errors live in Core `AccountErrors` (its single-source-of-truth directive) | Implemented | build-log/007 |
| ADR-041 | _(Persistence-001)_ `OpHaloDbContext` lives in **Foundation.Infrastructure** (DbSets for `Account`/`AccountUser`/`User`/`AccountEntitlements`; `ApplyConfigurationsFromAssembly`; ctor takes SharedKernel `IClock`). Two legacy behaviors ported **verbatim**: `SaveChangesAsync` timestamp interception (Added ⇒ `CreatedAtUtc`+`UpdatedAtUtc`; Modified ⇒ `UpdatedAtUtc`, via `IClock`) and the soft-delete **global query filters** over every `BaseEntity` — **except `AccountEntitlements`** (foundational satellite, ADR-025, always visible). The legacy `SystemRecord` / non-`BaseEntity` SaveChanges branches are dropped (not present here). `BaseEntityConfiguration` ports verbatim: key `ValueGeneratedNever` (domain mints v7 GUIDs), enums `HasConversion<string>().HasMaxLength(50)`. Design-time `OpHaloDbContextFactory` reads its own config (user-secrets→env→appsettings); snake_case; migrations history `__OpHaloMigrationsHistory`; `SystemClock` ported into Infrastructure on the single SharedKernel `IClock`. DI/host wiring + `appsettings` connection string **deferred to Phase 5** (no caller yet) | Implemented | build-log/008 |
| ADR-042 | _(Persistence-002)_ Three deliberate divergences from the legacy config: **(1)** `AccountUser.IsActive` is `builder.Ignore`d, not a column — it is a computed projection of `MembershipStatus == Active` (ADR-023); **(2)** primary ownership is a **nullable FK** `Account.PrimaryOwnerAccountUserId → AccountUser` (`OnDelete.Restrict`, no navigation), replacing the legacy `is_primary_owner` flag + `ix_account_users_one_primary_owner` filtered unique index (both dropped, ADR-019) — nullable owner FK + `Restrict` resolves the **circular FK** (`Account↔AccountUser`) so the migration creates `account_users`, then `accounts` with its owner FK inline, then adds `account_users.account_id → accounts` as a separate `AddForeignKey`; **(3)** migrations history table `__OpHaloMigrationsHistory`. Trimmed legacy columns (quotas/credits, halo booleans, Stripe/billing/delinquency, slug + intake token, `Account.Email`, push/welcome/install-prompt, `ProvisioningKey`, pilot dates) are absent because the entities were already trimmed in Phase 4a/4b — not reintroduced. Migration `InitialFoundationSchema` generated + inspected; `database update` against a live DB is the operator step | Implemented | build-log/008 |
| ADR-043 | _(Persistence-003)_ Phase 6B persistence proof tests use **Testcontainers.PostgreSql** against real PostgreSQL, not EF InMemory or SQLite. The proof target is provider/database behavior: migration application, partial/filtered unique indexes, FK enforcement around the Account↔AccountUser cycle, Npgsql duplicate-key errors, snake_case DDL, and the `AccountEntitlements` soft-delete filter exemption. Tests construct `OpHaloDbContext` directly with Npgsql + snake_case + `__OpHaloMigrationsHistory`; no repositories, `IUnitOfWork`, runtime DI, appsettings wiring, or host registration are introduced in this slice. Use the existing xUnit v2 lifecycle manually (`IAsyncLifetime`) rather than adding a Testcontainers xUnit integration package | Implemented | build-log/009 |
| ADR-044 | _(Persistence-004)_ Persisting the provisioning graph **requires two-phase save** due to the circular FK (Account↔AccountUser, ADR-019/042): **(1)** insert `User`, `Account` (with `primary_owner_account_user_id = NULL`), `AccountUser`, `AccountEntitlements` — EF cannot topologically order mutually-referencing inserts; **(2)** update `Account.primary_owner_account_user_id` to the owner's id. This is the **canonical persistence contract** for provisioning, not a test workaround. The real persistence boundary (Phase 6 repository/unit-of-work) must encapsulate this two-phase save; callers must never manage it manually. No domain or schema change required. Discovered and proven in the Phase 6B integration tests | Implemented | build-log/010 |

| ADR-046 | _(Keep-002)_ Public intake identity is a **Keep-owned hashed token satellite**, not columns resurrected on `Account`.

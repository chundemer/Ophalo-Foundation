# Decision Index — OpHalo Foundation

The living ledger of architecture/product decisions for **this** build. One line per
decision. This file is authoritative for the new project.

> The legacy reference app has its own `decision-index` and `decisions/` — those
> contain many decisions that no longer apply and are **not** used here. Consult
> them only as historical reference (`_reference/docs/`), and only after validation.

## Conventions

- Add a one-line entry the moment a decision is made. Expand to a full ADR under
  `docs/decisions/ADR-NNN-*.md` only when the rationale needs more than a line.
- IDs are sequential and never reused. A reversed decision is marked **Superseded**
  (link the successor), not deleted.

**Status legend:** `Implemented` = decided and built/verified in this repo ·
`Locked` = decided (build plan §4), not yet built · `Proposed` = under discussion ·
`Superseded` = replaced.

| ADR-031 | _(Phase4c-001)_ Slice scope = **§4.8 permission-key layer only**; split from §4.11 feature keys/limits (→ Phase 4d). **Permission key** (can this *user* do this *action*?) and **feature key** (is this *account* allowed this *capability*?) are deliberately distinct catalogs that compose later, never collapse. `keep.*` admitted into 4c only when it is a user action (`keep.requests.respond` ✓, `keep.enabled` ✗) | Implemented | build-log/005 |
| ADR-032 | _(Phase4c-002)_ Permission keys use a hierarchical `domain.resource.action` convention in a Foundation-owned string catalog (`PermissionKeys`) — adopted over the plan's flat-verb examples. | ADR-033 | _(Phase4c-003)_ Role hierarchy Owner ⊇ Admin ⊇ Operator ⊇ Viewer is expressed via **explicit set composition**, never enum numeric comparison — `AccountUserRole` values are identity, not authorization arithmetic. Two maps: `Base` (`account.*`/`keep.*`, every account) + `Internal` (`internal.*`, Internal-purpose accounts only, same hierarchy) | Implemented | build-log/005 |
| ADR-034 | _(Phase4c-004)_ `UserAccessPolicy.IsPermitted(role, membershipStatus, accountPurpose, key)` composes three gates: (1) only `MembershipStatus.Active` is permitted anything; (2) `internal.*` keys additionally require `AccountPurpose.Internal` — preserving the legacy `AdminGuard` semantic that internal authority comes from the account's purpose boundary, not a role name; (3) explicit role-map check. **Fail-closed**: empty/whitespace/unknown/non-canonical keys denied, no trimming. Matrix locks: `account.billing.manage` Owner-only; Operator gets `keep.insights.view`; internal support/platform keys deferred | Implemented | build-log/005 |
| ADR-035 | _(Phase4d-001)_ Slice scope = **§4.11 feature/limit layer**. Capabilities are **derived from the plan** via a static `PlanEntitlements` map ("plans produce entitlements"), **not** stored as per-account feature booleans on `AccountEntitlements` — that would recreate the legacy halo-boolean model dropped in ADR-028. The account's durable entitlement state stays compact (Plan/CommercialState/OperatingMode/MaxUserSeats/IsPilot); runtime authorizes against feature keys, never plan names (ADR-009) | Implemented | build-log/006 |
| ADR-036 | _(Phase4d-002)_ Two **separate** entitlement catalogs: `FeatureKeys` (boolean capability — "is it enabled?", `domain.capability`) and `FeatureLimitKeys` (numeric allowance — "what is the allowance?", `domain.x_limit`). Both distinct from `PermissionKeys` (4c) — feature/limit = account is entitled, permission = user is permitted; they compose, never collapse. `keep.active_request_limit` (a §4.11 example) is **deferred** — only `account.user_limit` + `keep.monthly_request_limit` are in v1 locked scope | Implemented | build-log/006 |
| ADR-037 | _(Phase4d-003)_ Usage limits resolve as **plan default with per-account override**; in v1 only `account.user_limit` is overridable, via the existing stored `AccountEntitlements.MaxUserSeats` (`> 0` wins, else plan default). No new per-account limit storage is added. "Unlimited" is the `int.MaxValue` sentinel (`FeatureLimitKeys.Unlimited`) so the natural `count < limit` call-site comparison needs no special-casing | Implemented | build-log/006 |
| ADR-038 | _(Phase4d-004)_ `FeatureAccessPolicy` is **fail-closed** (blank/unknown feature key → `false`; blank/unknown key or unmapped plan → limit `0`; exact match, no trimming) and **standalone** — no combined permitted+entitled facade until a real caller exists (composition at the call site: account allowed **and** user permitted **and** account entitled). The provisional tier matrix and limit values are intentionally **NOT locked** (§4.11): Trial is full-featured for evaluation, Starter omits mobile push + insights, limits constrain usage | Implemented | build-log/006 |
| ADR-039 | _(Provisioning-001)_ Account-creation orchestration is a **pure domain provisioning service** (`AccountProvisioningService` in Foundation.Application) — composes the four Phase 4 factories into the canonical `User + Account + owner AccountUser + AccountEntitlements` graph and returns it as `Result<AccountProvisioningResult>`. **No repositories / `IUnitOfWork` / MediatR / command-handler / DI** are introduced: the slice answers "what graph should exist?", and the persistence seam (save contract, transaction, unique constraints, aggregate boundary) is deferred to the persistence phase to avoid early-abstraction churn at that exact seam. Uses `DateTime` (UTC) to match the entity factory contracts, not the sketch's `DateTimeOffset` | Implemented | build-log/007 |
| ADR-040 | _(Provisioning-002)_ Cross-aggregate provisioning rules are **fail-closed**: the internal-account triad (purpose `Internal` ⇒ plan `Internal`, no pilot, no trial window) and the non-internal trial-window requirement return `Result.Failure`; seat allowance is **plan-derived** (`PlanEntitlements.LimitDefaults[plan][account.user_limit]`, ADR-037) — the method takes **no** seat parameter, so account↔entitlements seat drift is unrepresentable. Per-field validity (non-empty name, UTC timestamps, IANA TZ) is trusted to the entity factories (which throw); `Result.Failure` is reserved for composition rules. Primary owner is assigned in-memory pre-persistence (IDs exist at construction, ADR-019/024) and its failure is propagated. Provisioning errors live in Core `AccountErrors` (its single-source-of-truth directive) | Implemented | build-log/007 |
